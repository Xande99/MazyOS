## [2026-07-20] CSP nativo do Astro não ativado — conflita com View Transitions (ClientRouter)

**Decisão:** Não ativar `security: { csp: true }` no `astro.config.mjs` deste starter, apesar de ser o item 2.1 do plano de hardening (`MazyOS/AUDITORIA-SEGURANCA.md`, Fase 2). Fica registrado aqui como investigação futura, não como tarefa pendente da Fase 2.
**Motivo:** A documentação oficial do Astro (verificada via Context7 antes de qualquer alteração, 2026-07-20) declara `security.csp` incompatível com `<ClientRouter />`: *"Astro's enhanced view transitions via the ClientRouter component are also unsupported"* quando CSP está ativo. Este starter já usa `<ClientRouter />` em `src/layouts/Layout.astro` — é o nível 2 da hierarquia oficial de animação do `projeto/CLAUDE.md` ("faz o site parecer app na navegação"), etapa obrigatória do "Setup automático" de todo projeto Tipo A. Ativar CSP quebraria essa decisão arquitetural já em produção nos projetos nascidos deste starter.
**Alternativas descartadas:** Ativar CSP e remover `<ClientRouter />` — rejeitada sem consulta explícita ao usuário, por reverter uma decisão de UX/animação já validada em vez de ser um ajuste isolado de segurança. Resolvida por perguntar ao usuário em vez de decidir sozinho (ver AskUserQuestion na sessão de 2026-07-20): optou por pular o item agora.
**Impacto:** Nenhuma mudança de código. Mitigação de superfície de ataque de CDN de terceiros fica só a cargo do item 2.2 (self-host GSAP/Lucide), que reduz o risco original (CDN sem SRI) independente de CSP. Se o Astro em versão futura permitir CSP + ClientRouter juntos, ou se a decisão de manter View Transitions for revista, revisitar este item.

## [2026-07-22] --color-accent (vermelho) usado para CTA, --color-brand (verde) reservado pra identidade/links

**Decisão:** `--color-brand-*` recebeu o verde de marca (`#29C584`) e `--color-accent-*` (extensão opcional do contrato) recebeu o vermelho (`#E3233E`). Botões de CTA/conversão devem usar `--color-accent-600`, não `--color-brand` — desvio do texto literal do contrato (`_memoria/tokens-contract.md`, 1.2), que descreve `--color-brand` como "cor de marca primária (CTA, links, destaques)" em um único token.
**Motivo:** O cliente já definiu, na própria identidade visual consolidada, papéis distintos pras duas cores: verde é a cor de marca (logo, identidade), vermelho é especificamente "destaque/CTA". Forçar CTA a usar `--color-brand` (verde) ignoraria uma decisão de marca já tomada pelo cliente antes deste projeto existir.
**Alternativas descartadas:** Usar só `--color-brand` pro verde e não usar o vermelho em lugar nenhum do sistema de tokens (deixaria a cor de CTA do cliente fora do design system, forçando hex solto em componentes — proibido pelo lint de tokens). Renomear `--color-brand` para o vermelho e perder a associação verde=identidade — rejeitada por contrariar a associação visual que o cliente já usa (logo, Instagram).
**Impacto:** Qualquer seção com CTA (botão de compra, formulário) deve referenciar `var(--color-accent-700)` (ver correção de contraste abaixo, 2026-07-22) ou classes Tailwind geradas a partir dele — nunca `--color-brand` pra esse propósito. `--color-error` também mapeado pra essa família de vermelho, uso semântico consistente. Vale só pra este projeto — não é mudança no contrato global de tokens.

## [2026-07-22] Correção de contraste WCAG AA — verde e vermelho da marca precisam de degraus diferentes por contexto

**Decisão:** `--color-brand` (semântico) foi redefinido de `var(--color-brand-500)` pra `var(--color-brand-800)`. Todo uso de `bg-accent-600`/`text-accent-600` como CTA ou texto sobre superfície clara foi trocado pra `-700` (CTA sobre branco) ou `-800` (preço dentro do badge rosa, `--color-surface-alt`). As duas exceções que continuam usando o verde vibrante original (`text-brand-500` explícito, primitivo) são os textos verdes de `CTAFinal.astro`, que ficam sobre fundo escuro (`--color-surface-inverse`).
**Motivo:** Rodei o `mcp-accessibility-scanner` (axe-core) na página renderizada e ele confirmou, com número, o que a régua de contraste (calculado à parte com a fórmula de luminância relativa do WCAG) já apontava: o hex exato do cliente pro verde (`#29C584`, `brand-500`) mede 2.17:1 contra branco/texto branco — bem abaixo dos 4.5:1 exigidos pra texto normal ou 3:1 pra texto grande. O vermelho (`#E3233E`, `accent-600`) mede 4.45:1 contra branco — fica a menos de um centésimo do limite de 4.5:1, falha por uma margem mínima mas real. Isso não é um detalhe cosmético: é a diferença entre a página passar ou não no gate de acessibilidade do pipeline de QA (`Accessibility = 100`, `MazyOS/CLAUDE.md`), que é bloqueante pra considerar a entrega pronta.
**Alternativas descartadas:** Manter os hex exatos do cliente em todo lugar e aceitar a falha de contraste — rejeitada, viola o gate de acessibilidade não-negociável do pipeline de QA. Trocar as cores da marca por outras totalmente diferentes — rejeitada, o cliente já tem identidade visual consolidada (logo, Instagram, LP anterior); a solução certa é usar degraus mais escuros da mesma escala (mesmo hue/matiz), não trocar a cor.
**Impacto:** `brand-500` e `accent-600` continuam existindo como primitivos e são os valores certos pra uso *decorativo* (fundo de bloco sólido, imagem, ícone grande) ou como texto sobre fundo ESCURO — nunca como texto ou fundo-com-texto sobre superfície CLARA. Qualquer seção nova deste projeto que for adicionar texto colorido de marca sobre fundo claro deve usar `brand-800`/`accent-700`/`accent-800` (conforme o fundo específico — branco vs. rosa de suporte pedem degraus diferentes, ver comentário em `theme.css`), nunca o hex/primitivo -500/-600 direto. Medido e confirmado com scan real do axe-core antes e depois da correção — não estimado.

## [2026-07-22] `build.inlineStylesheets: 'always'` — CSS compilado inteiro embutido no HTML

**Decisão:** Setar `build.inlineStylesheets: 'always'` no `astro.config.mjs`, em vez do default `'auto'` do Astro (só inlina arquivo <4kb).
**Motivo:** Trace de performance real via Chrome DevTools MCP (mobile emulado, CPU 4x + Fast 4G, 2026-07-22) apontou o CSS compilado da página (Tailwind + `theme.css` + `@font-face` da Dancing Script, ~22kb) como request render-blocking, custando ~163ms de LCP/FCP — o arquivo passa do limite de 4kb do modo `'auto'` e vira um `<link>` externo síncrono em vez de inlinar. Como a LP é página única (sem rota de navegação interna ainda), não existe cache cross-page a perder — o trade-off de payload HTML maior compensa eliminar o round-trip de rede inteiro.
**Alternativas descartadas:** Dividir o CSS pra ficar sob 4kb (ex: mover `@font-face` pra outro arquivo) — mais complexo pra um ganho equivalente, e ainda deixaria pelo menos um link render-blocking. Ignorar os 163ms — rejeitado, é ganho de graça (mudança de 1 linha de config) que empurra a LCP mobile ainda mais fundo na faixa "good" (<2.5s).
**Impacto:** Se o projeto um dia ganhar navegação entre páginas (ex: página de política de privacidade), reavaliar — inlinar tudo deixa de valer a pena quando o cache de CSS entre páginas passa a importar. Nenhuma mudança de código além do config.

## [2026-07-22] Fontes da marca: Candara via system stack, Winter Christmas pendente de licença

**Decisão:** `--font-body` aponta pra `"Candara", "Segoe UI", system-ui, sans-serif` (sem arquivo embutido). `--font-display` aponta pra `"Winter Christmas", cursive` (nome referenciado, sem `@font-face`, sem arquivo).
**Motivo:** Candara é fonte proprietária da Microsoft (distribuída com Windows/Office) — não pode ser self-hosted como web font sem licença própria pra isso; usá-la via system font stack é legal e funciona pra quem tem Windows/Office, com fallback gracioso pro resto. Winter Christmas é uma fonte script cuja licença de uso web/comercial o cliente ainda não confirmou — a maioria das fontes script gratuitas (ex: dafont.com) é só pra uso pessoal.
**Alternativas descartadas:** Embutir os arquivos de qualquer uma das duas fontes sem confirmação de licença — rejeitado por risco legal (uso comercial de fonte sem licença adequada).
**Impacto:** `--font-display` renderiza com o fallback `cursive` genérico do navegador até a licença de Winter Christmas ser resolvida (ver pendência em `clientes/Cozinha e Tal/briefing.md`) — visual do hero/títulos não é o final até isso ser corrigido. Não bloqueia desenvolvimento de estrutura/copy da LP, só o polish tipográfico final.
