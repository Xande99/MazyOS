## [2026-07-09] Animação de scroll da seção "solução": GSAP + ScrollTrigger via CDN, não bundler

**Decisão:** Seção `#solucao` (etapas do processo) ganhou uma timeline pinada (`scrollTrigger:{pin:true,scrub:0.6}`) com barra de progresso via `transform:scaleX()` e 4 etapas ativando em sequência (classe `.step--active`). GSAP 3.15.0 + ScrollTrigger carregados via `<script>` do unpkg, no mesmo padrão do Lucide já usado no site — não via npm/bundler.
**Motivo:** `duPolvoNovo/duPolvo/` é um site estático vanilla JS (sem build step, sem `package.json`), então CDN é a única forma de trazer o GSAP sem reestruturar o projeto. A stack oficial de Tipo A (`projeto/CLAUDE.md`) prevê Astro + GSAP via npm — esse site ainda não migrou pra Astro (ver pendência registrada no `CLAUDE.md` raiz), então a decisão aqui é uma ponte até essa migração acontecer, não o padrão final.
**Alternativas descartadas:** Animar a linha de progresso com SVG `stroke-dashoffset` (efeito de "desenhar" mais literal) — rejeitada por violar a regra de só animar `transform`/`opacity` (stroke-dashoffset tem custo de repaint, não é compositor-only). Também descartado usar `gsap.to()` com `transform:scale` direto no `.step__ic` teria conflitado com o hover em CSS (`transform` inline do GSAP sobrepõe a transição de hover) — resolvido usando GSAP só pra pin/scrub/timeline, e classes CSS (`.step--active`) pro estado visual de cada etapa, mantendo o hover intacto.
**Impacto:** Estado inicial "escondido" dos ícones/textos só existe atrás da classe `.motion-ready`, adicionada via JS somente quando GSAP+ScrollTrigger carregam e `prefers-reduced-motion` não está ativo — sem JS ou com motion reduzido, a seção sempre mostra o estado final, sem tela vazia. Ainda não verificado com Playwright MCP (pendência separada, não é decisão em aberto).
