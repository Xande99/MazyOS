## [2026-07-20] CSP nativo do Astro não ativado — conflita com View Transitions (ClientRouter)

**Decisão:** Não ativar `security: { csp: true }` no `astro.config.mjs` deste starter, apesar de ser o item 2.1 do plano de hardening (`MazyOS/AUDITORIA-SEGURANCA.md`, Fase 2). Fica registrado aqui como investigação futura, não como tarefa pendente da Fase 2.
**Motivo:** A documentação oficial do Astro (verificada via Context7 antes de qualquer alteração, 2026-07-20) declara `security.csp` incompatível com `<ClientRouter />`: *"Astro's enhanced view transitions via the ClientRouter component are also unsupported"* quando CSP está ativo. Este starter já usa `<ClientRouter />` em `src/layouts/Layout.astro` — é o nível 2 da hierarquia oficial de animação do `projeto/CLAUDE.md` ("faz o site parecer app na navegação"), etapa obrigatória do "Setup automático" de todo projeto Tipo A. Ativar CSP quebraria essa decisão arquitetural já em produção nos projetos nascidos deste starter.
**Alternativas descartadas:** Ativar CSP e remover `<ClientRouter />` — rejeitada sem consulta explícita ao usuário, por reverter uma decisão de UX/animação já validada em vez de ser um ajuste isolado de segurança. Resolvida por perguntar ao usuário em vez de decidir sozinho (ver AskUserQuestion na sessão de 2026-07-20): optou por pular o item agora.
**Impacto:** Nenhuma mudança de código. Mitigação de superfície de ataque de CDN de terceiros fica só a cargo do item 2.2 (self-host GSAP/Lucide), que reduz o risco original (CDN sem SRI) independente de CSP. Se o Astro em versão futura permitir CSP + ClientRouter juntos, ou se a decisão de manter View Transitions for revista, revisitar este item.

## [2026-07-23] Hover do `GridOferta.astro` corrigido: `box-shadow` → `transform`

**Decisão:** Em `src/sections/GridOferta.astro`, o hover de elevação do card trocou de `transition-shadow duration-300 hover:shadow-md` para `transition-transform duration-300 hover:-translate-y-1!` (sufixo `!important` do Tailwind v4).
**Motivo:** Achado replicado do projeto Portfolio (nascido deste mesmo starter): `box-shadow` não é `transform`/`opacity`, violando a regra não negociável do `projeto/CLAUDE.md` ("Animar somente transform e opacity"). O sufixo `!` é necessário porque o `<article>` carrega a classe `.reveal` (sistema de reveal-on-scroll, `src/scripts/reveal.ts` + `src/sections/motion.ts`) — o GSAP deixa um `transform` inline no elemento como estado final da animação de entrada, e esse inline style vence uma classe Tailwind normal de hover baseada em `transform`. Sem o `!`, o hover ficaria silenciosamente inerte. Ver a mesma decisão, com a medição de contraste/comportamento original, em `projeto/Portfolio/.claude/decisions.md` (entrada de 2026-07-23).
**Alternativas descartadas:** Corrigir na raiz (fazendo `motion.ts`/`reveal.ts` limpar a propriedade de transform inline via `clearProps` ao final da timeline) — descartada pelo mesmo motivo já documentado no Portfolio: quebraria o estado inicial oculto do `.reveal` em todas as seções de todo projeto nascido deste starter, mudança grande demais pra uma correção pontual de card.
**Impacto:** Todo projeto novo que nascer deste starter a partir de agora já herda o hover corrigido. Projetos já nascidos ANTES desta correção (ex: clientes ativos que já usam este starter) mantêm o bug antigo em qualquer seção que tenha copiado `GridOferta.astro` — não corrigido retroativamente aqui, avaliar caso a caso se vale a pena portar a correção pra esses projetos já entregues.
**Nota de verificação:** `npm run lint:css` passou limpo. `npm run check`/`build` não puderam ser executados neste ambiente — uma política de Controle de Aplicativo do Windows bloqueou o binário nativo do compilador do Astro (`@astrojs/compiler-binding-win32-x64-msvc`, erro `ERR_DLOPEN_FAILED`, "Uma política de Controle de Aplicativo bloqueou este arquivo"), problema de ambiente/máquina não relacionado a esta mudança. A sintaxe da correção é idêntica à já validada com build real em `projeto/Portfolio/src/components/ProjetoCard.astro` (Task 3, revisão + fix + re-revisão aprovados). Recomendado rodar `npm run build` neste starter fora deste ambiente sandbox pra confirmar antes do próximo `/novo-projeto` que consumir este template.
